# model/

The two training stages. They never talk to each other in code — only through files in
`artifacts/`.

| Stage | What it does |
|---|---|
| [`retriever/`](retriever/) | Two-tower (PyTorch). Embeds users + all ~22.8k anime into one 128-d cosine space → top-K candidates. |
| [`ranker/`](ranker/) | LightGBM lambdarank. Reranks the retriever's top 200 on 29 features. |

## The artifacts contract

`artifacts/` (repo root) is the firewall: **retriever writes, ranker + service read.** The ranker
and service may import model *definitions* (e.g. `UserTower`) but never training code, and only
the ranker's own two files (`ranker.txt`, `ranker_meta.json`) are written by anything other than
the retriever. Full schema — shapes, dtypes, provenance — lives in `artifacts/CONTRACT.md`,
generated by `retriever/export.py`. Don't hand-edit it.

This is what lets the service stay untouched across retrains: new model, same file names, same
shapes.

## Retrain loop

Run in order; each step's output feeds the next.

```bash
# 1. retriever: data/cleaned → train-data → best.pt → artifacts/
for s in model/retriever/data_prep/0[1-6]_*.py; do venv/bin/python "$s"; done
venv/bin/python model/retriever/data_prep/99_verify.py
#    train on Colab (retriever/train.ipynb), download best.pt into retriever/checkpoints/
venv/bin/python model/retriever/export.py && venv/bin/python model/retriever/test_export.py

# 2. ranker: artifacts → pools → model → artifacts/
venv/bin/python model/ranker/data_prep/build_eval.py
venv/bin/python model/ranker/eval.py --baseline-only     # sanity gate — must pass before training
venv/bin/python model/ranker/data_prep/build_train.py
venv/bin/python model/ranker/src/train_lgbm.py
venv/bin/python model/ranker/eval.py                     # select winner
venv/bin/python model/ranker/export.py
```

Training location: retriever needs a GPU (Colab); the ranker trains locally on CPU (~40 min).

## Verify

```bash
venv/bin/python -m pytest model/retriever/tests -q     # 31 tests
venv/bin/python -m pytest model/ranker/tests -q        # 16 tests
```

Run the two suites **separately** — both import flat and define modules named `config`/`metrics`,
so a combined run collides.
